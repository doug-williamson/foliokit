## FINDING-01 — InputSignal brand mismatch with separate Angular node_modules instances
**Severity: Library bug (real-world edge case)**
**Status: Workaround applied in smoke-test tsconfig**

### What happened

`ng build` failed with:

```
TS2551: Property '__@ɵINPUT_SIGNAL_BRAND_WRITE_TYPE@2478' does not exist on type
'InputSignal<string>'. Did you mean '__@ɵINPUT_SIGNAL_BRAND_WRITE_TYPE@978'?
```

### Root cause

TypeScript `unique symbol` types are unique per **file-system path**, not per package version. The FolioKit dist libraries were compiled against `@angular/core` at:

```
/home/user/foliokit/node_modules/@angular/core
```

The smoke-test project has its own Angular at:

```
/home/user/foliokit/smoke-test/node_modules/@angular/core
```

Even though both installed `21.2.4`, they are distinct files on disk → distinct unique symbol IDs → type incompatibility in Angular's template type checker for `InputSignal`.

This issue manifests only with local `file:` installs where the consumer and the build environment share a filesystem but have separate `node_modules` trees.

### When this does NOT occur

- Normal npm/yarn install of published packages: the consumer's `node_modules` contains one Angular instance; the library's `.d.ts` files resolve `@angular/core` from that same instance → compatible symbols.
- Within the monorepo itself: path aliases in `tsconfig.base.json` redirect library imports to source files, bypassing `.d.ts` entirely.

### Workaround (applied)

`smoke-test/tsconfig.json` `paths` redirect all `@angular/*` to the monorepo's node_modules (the Angular version used to build the dist):

```json
"paths": {
  "@angular/core": ["../node_modules/@angular/core"],
  "@angular/core/*": ["../node_modules/@angular/core/*"],
  ...
}
```

### Decision needed

This should not affect real npm consumers, but it is worth verifying once the packages are published to npm and installed in a fresh project. If it does surface, the mitigation is to ensure dist libraries are rebuilt against the same Angular minor.minor.patch that consumers are expected to use and document the required version range explicitly.

## FINDING-02 — Initial bundle size budget exceeded
**Severity: Warning only (not an error)**
**Status: No action required — adjust budgets per project**

```
▲ bundle initial exceeded maximum budget. Budget 500kB was not met by 424kB with a total of 924kB.
```

The smoke test's `angular.json` has the default 500 kB budget generated by `ng new`. The full FolioKit dependency set (Angular Material + Firebase SDK + ngx-markdown) produces an initial bundle of ~924 kB. This is expected for a feature-complete CMS application.

**Action for real apps:** Raise or remove the initial budget in `angular.json`, or implement lazy-loaded routes for admin pages.

## FINDING-03 — `tokens.css` asset path includes `src/` prefix
**Severity: Cosmetic / DX concern**
**Status: Functional, flagged for your decision**

The ng-package.json assets entry:

```json
"assets": ["src/styles/tokens.css", "src/styles/tokens.scss"]
```

Causes ng-packagr to copy assets preserving their relative paths. After `dist/` output (and npm publish), consumers must import via:

```json
"styles": ["node_modules/@foliokit/cms-ui/src/styles/tokens.css"]
```

The `src/` prefix is unconventional — most published libraries put assets at the package root or in a flat `styles/` directory. This is functional but may surprise consumers.

**Decision needed:** Flatten the asset path? Change `ng-package.json` to declare assets from the `src/styles/` directory as a glob, and configure `destDir`:

```json
"assets": [
  { "include": "tokens.css", "input": "src/styles", "output": "styles" },
  { "include": "tokens.scss", "input": "src/styles", "output": "styles" }
]
```

This would produce `node_modules/@foliokit/cms-ui/styles/tokens.css` — much cleaner. Left as-is until you decide, since changing it is a minor breaking change to the documented path in `README.md`.

## FINDING-04 — Function name: `provideFolioKit()` vs `providesFolioKit()`
**Severity: Naming decision**
**Status: Implemented as `provideFolioKit()` — confirm or rename**

The task specified `providesFolioKit()` (with a trailing 's'). Angular's convention for all provider helpers is `provide*()` without 's' (e.g., `provideRouter`, `provideFirebase`, `provideHttpClient`). The implementation uses `provideFolioKit()`.

**Decision needed:** Accept `provideFolioKit()` or rename to `providesFolioKit()`?

## FINDING-05 — `SITE_ID` token has no current consumers
**Severity: Informational**
**Status: No action required**

`SITE_ID` is a new injection token created in `foliokit.providers.ts`. It is populated when `siteId` is provided to `provideFolioKit()`. No existing service or component in any FolioKit library currently injects it. It is a forward-looking API for multi-site deployments.

## FINDING-06 — Angular Material theme must be declared by the consumer
**Severity: Known limitation, documented**
**Status: No fix needed — working as intended**

`AppShellComponent` from `@foliokit/cms-ui` uses Material components (sidenav, toolbar, buttons). Angular Material's M3 theme (`@include mat.theme(...)`) cannot be bundled inside a library due to ng-packagr constraints — it must be called in the consumer's global stylesheet.

The smoke test's `src/styles.scss` demonstrates the required setup. This is documented in `libs/cms-ui/README.md`.
